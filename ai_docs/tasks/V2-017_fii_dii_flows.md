## The 6 Locked Decisions
*Resolved in conversation with Lijo (2026-05-19) after Gemini-agent source recon. Do not re-derive.*

### Decision 1 — V1 primary source: **Moneycontrol `__NEXT_DATA__` parse**
`GET https://www.moneycontrol.com/markets/fii-dii-data/cash/` → extract the
`<script id="__NEXT_DATA__" type="application/json">` blob from the raw HTML →
`JSON.parse` → read `props.pageProps.FiiDiiData.fiiDiiData[]`.

**Why**: It delivers the **same-day signal AND the deepest available window
(30 trading days) in one build-ID-free request**, headless (no cookies/headers
needed — confirmed). NSE = current day only; BSE = per-date and DII-only. For
the *trading* consumer, Moneycontrol's same-day FII+DII net is exactly what's
needed. Deep history (for Product C) is a separate problem — see Decision 3.

**Implementation hint**: **Never hardcode the `_next/data/<buildId>/FiiDii.json`
URL** — `buildId` rotates on every Moneycontrol deploy and silently 404s. Always
parse the inline `__NEXT_DATA__` script from the page HTML. One request, no buildId.

### Decision 2 — Cron model: **separate daily script + separate Railway cron**
`scripts/seed-india-flows.mjs`, its own Railway cron, scheduled **once daily after
market close** (≈ 19:30 IST / 14:00 UTC; tune after observing publish time).

**Why**: Failure isolation — reuses V2-013 Decision 3's philosophy. FII/DII is one
row per trading day; running it on the 10-min news cron means ~143 no-op runs/day
and couples a flows-scrape failure to the news pipeline. A separate cron keeps the
two blast radii disjoint.

**Implementation hint**: Still use `runSeed()` for the lock + freshness-meta +
exit-0 contract, but the canonical Redis key is a *status* key only
(`news:flows:v1:india`) — the **data** goes to PostgreSQL, not Redis. This script
does NOT read `news:digest:v1:india:en`.

### Decision 3 — Backfill: **30-day seed now; deep history is V2-017b (parallel, non-blocking)**
*Revised 2026-05-19 after ground-truth recon disproved the "deep history in one
payload" assumption. Moneycontrol carries only 30 trading days; NSE 1 day; BSE
per-date DII-only. No deep FII/DII archive exists in the recon.*

`scripts/backfill-india-flows.mjs` walks the **30-trading-day** `fiiDiiData[]`
window and upserts every row (`source='moneycontrol'`, same `ON CONFLICT`
upsert as Decision 4). That is the entire V1 backfill.

**Why**: Two consumers, two needs. **Trading** needs the live forward signal —
V2-017's daily pipeline + 30-day seed serves that *now*. **Product C (selling
the data)** needs large historical depth — but no source for that exists in the
recon, and hunting one must not block the trading pipeline. So Product C depth
**accrues forward** from deploy (+ the 30-day head start), and deep historical
acquisition is split into **V2-017b** (research-first; see below). This is
roadmap-aligned: trading proof-of-value first, dataset depth compounds over time.

**Implementation hint**: V2-017's schema already future-proofs V2-017b at zero
cost — the `source` column + `ON CONFLICT DO UPDATE` upsert (Decision 4) lets
V2-017b retro-load any historical archive later with **no schema change**, just
`source='<archive-name>'`. Build V2-017 now; nothing for Product C is lost.

#### V2-017b — Deep FII/DII History (parallel research track, NOT this task)
- **Status**: research-first; build gated on finding a real deep archive.
  Runs in parallel (Gemini agent recon) while V2-017 is built/running.
- **Goal**: locate a true historical FII/DII source (NSDL FPI archive, NSE
  archives, NSDL/CDSL CSV dumps, SEBI bulletins, or a vendor) → one-time
  retro-load into `india_institutional_flows` via the existing upsert.
- **Not started here.** Task-file V2-017b only once the archive source is
  confirmed by recon (per the "task-filed = reality, not aspiration" rule).

### Decision 4 — Provenance: **`source` column + `ON CONFLICT DO UPDATE`**
Every row stores `source TEXT NOT NULL` (`'moneycontrol' | 'nse' | 'bse'`). Primary
key `(flow_date, investor_type, segment)`. Upsert is `ON CONFLICT DO UPDATE` so a
later, more-authoritative source (NSE/BSE) can **supersede** a Moneycontrol row with
**zero schema change** when the future superseding path ships.

**Why**: Audit-trail philosophy, same as V2-014's `sentiment_source`. The quant model
can down-weight aggregator rows; the Product C export can filter to exchange-sourced
rows; the upgrade path is non-breaking.

**Implementation hint**: Only supersede *upward* in trust (`moneycontrol` →
`nse`/`bse`), never downgrade. V1 only ever writes `moneycontrol`, so the guard is
a no-op in V1 but the column + upsert shape must exist now.

### Decision 5 — Holiday / no-publish days: **write nothing, log one line, never error**
Market holidays and pre-publish runs return no new row. The script logs a single
`[flows] no new rows for <date> (holiday or not yet published)` line and exits 0.
Never write a zero/placeholder row; never throw.

**Why**: A zero row is indistinguishable from a real "flat flows" day and would
poison the dataset. Silent skip + log is the only correct behaviour.

### Decision 6 — Prod-execution boundary
Per `memory/feedback_v2_prod_execution.md`: this task delivers **code + DDL +
read-only self-checks only**. **Lijo runs** `migrate-india-signals.mjs`, the daily
seed, and the backfill against prod after review. The agent does NOT run the scrape
or migration against prod, does NOT create/run `scratch/` probes (Gemini agent's job).

## Schema (Locked)

Add to `scripts/migrate-india-signals.mjs` (DDL runner — Lijo executes):

```sql
CREATE TABLE IF NOT EXISTS india_institutional_flows (
  flow_date      DATE NOT NULL,
  investor_type  TEXT NOT NULL,            -- 'FII' | 'DII'
  segment        TEXT NOT NULL DEFAULT 'cash',
  gross_buy      DECIMAL(14,2),            -- ₹ crore
  gross_sell     DECIMAL(14,2),            -- ₹ crore
  net            DECIMAL(14,2) NOT NULL,   -- ₹ crore (buy − sell)
  source         TEXT NOT NULL,            -- 'moneycontrol' | 'nse' | 'bse'  (provenance)
  is_provisional BOOLEAN NOT NULL DEFAULT TRUE,
  created_at     TIMESTAMPTZ DEFAULT NOW(),
  updated_at     TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (flow_date, investor_type, segment)
);
CREATE INDEX IF NOT EXISTS idx_flows_date        ON india_institutional_flows (flow_date DESC);
CREATE INDEX IF NOT EXISTS idx_flows_type_date   ON india_institutional_flows (investor_type, flow_date DESC);
```

Upsert shape (daily + backfill):
```sql
INSERT INTO india_institutional_flows
  (flow_date, investor_type, segment, gross_buy, gross_sell, net, source, is_provisional)
VALUES ($1,$2,'cash',$3,$4,$5,$6,TRUE)
ON CONFLICT (flow_date, investor_type, segment) DO UPDATE
  SET gross_buy = EXCLUDED.gross_buy,
      gross_sell = EXCLUDED.gross_sell,
      net = EXCLUDED.net,
      source = EXCLUDED.source,
      updated_at = NOW()
  WHERE india_institutional_flows.source = 'moneycontrol'   -- only supersede upward (Decision 4)
     OR EXCLUDED.source = india_institutional_flows.source;
```

## Pipeline Flow

```
Daily Railway cron (≈14:00 UTC) → runSeed('india','flows','news:flows:v1:india', fetchFlows)

fetchFlows():
  1. GET moneycontrol.com/markets/fii-dii-data/cash/  (Chrome UA + Accept header)
  2. Extract <script id="__NEXT_DATA__">…</script>  → JSON.parse
  3. arr = props.pageProps.FiiDiiData.fiiDiiData   (else: log + exit 0, Decision 5)
  4. Map each row → { flow_date, FII{gb,gs,net}, DII{gb,gs,net} }
     parse Moneycontrol date format → ISO DATE
  5. Filter to rows newer than MAX(flow_date) already in PostgreSQL
  6. If none → log "[flows] no new rows…" → return {written:0}   (Decision 5)
  7. For each new date: upsert FII row + DII row (source='moneycontrol')
  8. Return { written, latest_date } → runSeed publishes to status key

backfill-india-flows.mjs (one-time, Lijo runs):
  - same fetch + parse, upsert the full 30-trading-day window
    (~60 rows = 30 dates × {FII, DII}); idempotent via ON CONFLICT
  - log start + final count (volume is tiny — no chunk logging needed)
  - V2-017b will later reuse this script's upsert for deep archive retro-load
```

## Research Appendix — Source Recon (Gemini agent)
*Endpoints/headers are confirmed from the `scratch/` probe scripts. Runtime

### Moneycontrol (V1 PRIMARY)
- URL: `https://www.moneycontrol.com/markets/fii-dii-data/cash/`
- Mechanism: parse inline `<script id="__NEXT_DATA__" type="application/json">` →
  `props.pageProps.FiiDiiData.fiiDiiData[]`
- Headers: `User-Agent: <Chrome UA>`, `Accept: text/html,…`
- Build-ID trap: do NOT use `/_next/data/<buildId>/FiiDii.json` — buildId rotates
- Runtime HTTP status: **200** (confirmed, plain headless `fetch()`, **no
  cookies / no custom headers needed**)
- `__NEXT_DATA__` present headless?: **Yes** — hydration JSON in raw HTML
- Array length (= backfill depth): **30** — rolling last 30 *trading days* only.
  ⚠️ NOT deep history. This breaks the original Decision 3 assumption.
- Date format: **`YYYY-MM-DD`** (e.g. `"2026-05-18"`) — no date parsing needed
- Field names (note the `Sales` vs `Sale` asymmetry — match exactly):
  `date`, `fiiPurchase`, `fiiSales`, `fiiNet`, `diiPurchase`, `diiSale`, `diiNet`
- Value format: **comma-grouped strings**, negatives prefixed `-`
  (e.g. `"17,222.18"`, `"-587.80"`) → strip `,` then `parseFloat`
- Confirmed sample (18-May-2026): FII net **+2,813.69**, DII net **+2,682.12**

### NSE (future superseding path — Decision 4)
- Bootstrap: `GET https://www.nseindia.com/` → **200**, returns `set-cookie`
  (`nsit`, `nseappid`, …) — warm-up call is **required**
- Data: `GET https://www.nseindia.com/api/fiidiiTradeReact` with `Cookie` →
  **200**. Array length **2** (DII + FII/FPI), **1 unique date — current day
  only. Zero backfill from NSE** (confirmed).
- Date format: `DD-MMM-YYYY` (`"18-May-2026"`). Fields: `category`
  (`"DII"` / `"FII/FPI"`), `date`, `buyValue`, `sellValue`, `netValue`
  (plain numeric strings, no commas). Note label is `FII/FPI`, not `FII`.

### BSE (future superseding path — Decision 4)
- Working URL (confirmed): `https://api.bseindia.com/BseIndiaAPI/api/
  StockpricesearchData/w?MonthDate={DD/MM/YYYY}&Scode=&Seg=C&
  YearDate={DD/MM/YYYY}&pageType=1&rbType=D` (date in BOTH params)
- Required headers: Chrome UA, `Referer: https://www.bseindia.com/markets/
  equity/EQReports/categorywise_turnover`, `Origin: https://www.bseindia.com`
  (CORS-blocked without Referer+Origin)
- Response: `CatStockData[]` with `REPORTING_DATE`, `*_DII_BSENSE`, `*_CLIENT`,
  `*_NRI`, `*_OWN` (comma-grouped strings)
- ⚠️ **FII/FPI is NOT exposed** — only DII (`*_DII_BSENSE`) is populated.
  BSE is a **DII-only, per-date** superseding source. Cannot supersede FII.

## Error Scenarios

| Symptom | Likely cause | Fix |
|---|---|---|
| `No __NEXT_DATA__ script found` | Moneycontrol served bot/interstitial HTML to headless fetch | Add/adjust `Accept` + `User-Agent`; if persistent, escalate to NSE/BSE superseding path (Decision 4) |
| `fiiDiiData` undefined | Moneycontrol changed the prop path | Re-inspect `props.pageProps` keys; the Research Appendix records the expected path — update + note the drift |
| Hardcoded `_next/data/<id>` 404s | Used the buildId URL (forbidden) | Switch to inline `__NEXT_DATA__` parse (Decision 1 hint) |
| Duplicate-key error | Upsert not used | Use the `ON CONFLICT DO UPDATE` from Schema |
| Zero rows on a holiday but a row got written | Violated Decision 5 | Guard: only upsert when the source array actually contains the date |
| Cron overlaps | Missing lock | `runSeed()` provides `seed-lock:india:flows` — keep the `runSeed` wrapper |
