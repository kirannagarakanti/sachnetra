## Pattern To Follow

### JSON description parsing — from the render function in `setupDesktopIndiaLayout()`:

Currently, descriptions are rendered raw:
```typescript
<div class="sn-sidebar-finding-desc">${escapeHtml(f.description)}</div>
```

Add a sanitizer function BEFORE `escapeHtml()` that detects and parses JSON:
```typescript
const cleanDescription = (desc: string): string => {
  // If the description looks like JSON, try to extract summary text
  const trimmed = desc.trim();
  if (trimmed.startsWith('{') && trimmed.endsWith('}')) {
    try {
      const parsed = JSON.parse(trimmed);
      // Extract all .summary fields from the parsed object
      const summaries: string[] = [];
      for (const key of Object.keys(parsed)) {
        if (parsed[key]?.summary) summaries.push(parsed[key].summary);
      }
      if (summaries.length > 0) return summaries.join(' ');
    } catch { /* not valid JSON, use as-is */ }
  }
  return desc;
};
```

Then use: `escapeHtml(cleanDescription(f.description))`

## Session Notes — Lessons Learned for Future Agents

### Key decisions made

1. **Parse at display, not at source** — We fix the JSON at render time in the sidebar, not in the correlation service. This avoids touching the shared data pipeline and keeps the fix scoped to SachNetra. The correlation service may be used by all variants — never assume fixing it there is safe.

2. **The AI summarizer has 3+ JSON schemas, not one** — The initial implementation assumed a single schema (`{ summaryN: { summary: "string" } }`). Production data revealed at least 3 distinct shapes:
   - **Pattern A**: `{ "summary1": { "summary": "string" } }` — nested named keys
   - **Pattern B**: `{ "summary": "string", "meaning": "string" }` repeated as concatenated objects (not valid JSON as a whole)
   - **Pattern C**: `{ "summary": { "1": { "event": "string" } } }` — doubly nested numbered events
   Always use a **recursive extractor** with priority-key fallback, not a schema-specific parser. The AI will surprise you.

3. **Concatenated JSON objects are not valid JSON** — Pattern B produces `{...} {...}` which `JSON.parse` rejects. Fix: use a regex (`/\{(?:[^{}]|\{[^{}]*\})*\}/g`) to extract individual blocks and parse each one separately. This handles any number of concatenated objects.

4. **Always deduplicate extracted texts** — The AI sometimes returns the same summary text twice across repeated blocks (confirmed in "kerala" finding). Using `[...new Set(texts)]` before joining removes exact duplicates silently without edge cases.

5. **Use `\n` + `<br>` for line-by-line display, not `' • '`** — Joining with bullet character `•` puts all text on one long line. For multi-event findings (like "crore" with 4 events), use `\n` as the separator in JS, then split on `\n` and join with `<br>` when building the HTML template. Add `white-space: pre-line` on the `.expanded` state for proper rendering when unclamped.

6. **TypeScript strict mode: `Record<string, X>[key]` returns `X | undefined`** — Under `noUncheckedIndexedAccess`, index access on a `Record<string, unknown>` can return `undefined` even if the key is from `Object.keys()`. Always store the accessed value in a local variable and check `typeof val === 'string'` before using it as a string.

7. **Sidebar breakpoint was badly over-conservative** — The original rule hid the sidebar at `max-width: 1024px`, which meant it only appeared at >1024px. With a 380px sidebar + ~400px minimum for the river, the sidebar is comfortable from ~820px. Rule of thumb: hide breakpoint should be `sidebar_width + min_river_width`, not an arbitrary threshold like 1024px.

8. **Dead zone anti-pattern: desktop layout with no sidebar** — The range 769–820px had a desktop layout (two-column grid, chip bar, etc.) but the sidebar was hidden via the 1024px rule. This gives users a worse experience than mobile (no bottom nav, no brief card, but also no sidebar). The fix: make the mobile/desktop breakpoint match the sidebar-hide breakpoint, so there's never a "desktop without sidebar" state. Mobile handles ≤820px, desktop from 821px.

9. **Mobile hide rules need to cover new mobile-territory ranges** — When we expanded mobile from ≤768px to ≤820px, the elements that were hidden in the desktop block (`.pizzint-indicator`, `.community-widget`) became visible again in the 769–820px range because those hide rules are inside `@media (min-width: 821px)`. Always re-add hide rules to the mobile block for any WorldMonitor elements that are newly exposed.

10. **Font sizes at 10-11px are too small for sidebar content** — Findings descriptions at 10px and titles at 11px are hard to read in a sidebar context with real text. Use at minimum 12px for body text and 13px for titles in sidebar panels. Also increase line-height from 1.4 to 1.5 for readability and bump `line-clamp` from 2 to 3 when font gets larger.
