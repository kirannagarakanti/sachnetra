## The 9 Locked Decisions
*Resolved with Lijo (2026-05-21) after Gemini-agent source recon. Do not re-derive.*

### Decision 1 — Scope: **persist to the PostgreSQL quant layer; NO frontend/handler work**
The roadmap one-liner ("wire `get-bis-policy-rates.ts` to India queries") is misleading.
India BIS data is **already fetched** to Redis by the inherited `seed-bis-data.mjs` and
the handler already returns the India row. The real gap is **consumption by the quant
model**: nothing persists BIS India series to PostgreSQL. V2-020 closes that gap and
nothing else — no `src/`, no `api/`, no variant, no handler. The `economic: false`
map-layer flag in `india.ts` is a globe overlay toggle, unrelated to this pipeline; leave
it alone.

**Why**: V2 mission — "SachNetra is the collection engine; the database is the asset."
Block-2 consistency: V2-017/018/019 are all backend PG-layer collectors. V2-020 matches.

### Decision 2 — Source & format: **BIS SDMX REST API, CSV, fully-qualified keys**
`GET https://stats.bis.org/api/v1/data/<DATASET>/<KEY>?startPeriod=<P>&detail=dataonly&format=csv`
with a plain Chrome `User-Agent`. No cookies, no warm-up, no auth (confirmed Part E).

**Why**: SDMX is a clean, documented, public standard (the opposite of the NSE/RBI
scraping pain). `stats.bis.org/api/v1` is the canonical machine host; `data.bis.org` is
the human charts site and is NOT an API (Part E).

**Implementation hint**: We query **fully-qualified keys** (e.g. `WS_TC/Q.IN.P.A.M.770.A`),
so each request returns **exactly one series**. The parser therefore only needs the
`TIME_PERIOD` + `OBS_VALUE` columns — no positional dimension reading, no label-guard
(contrast V2-019). This makes the parser trivial and drift-resistant: the `series_code`
IS the full identity. Skip rows where `OBS_VALUE` is empty / `NaN`.

### Decision 3 — Cron model: **separate WEEKLY script + separate Railway cron**
`scripts/seed-india-macro.mjs`, its own Railway cron, scheduled **weekly**.

**Why**: These are monthly/quarterly macro series — a daily/10-min cron would be ~all
no-ops. Weekly catches a policy-rate change or new monthly print within a week while
staying cheap (8 small GETs). Separate cron = failure isolation (V2-017 Decision 2).
`runSeed()` provides the lock + freshness + exit-0.

**Implementation hint**: Canonical Redis key is a *status* key only
(`news:macro:v1:india`) — data → PostgreSQL. Does NOT read `news:digest:v1:india:en`.

### Decision 4 — Schema: **tall fact table + series-meta lookup**
A **tall** `india_macro_rates` (`series_code, time_period, obs_value, …`) keyed by
`(series_code, time_period)`, plus a small `india_macro_series_meta` lookup that maps each
`series_code` → human label, unit, frequency, revises-flag.

**Why — diverges from V2-019's wide table (intentional)**: V2-019 (RBI WSS) is wide
because it's one release, one date, a fixed indicator set. BIS is the opposite — many
independent long series, mixed frequencies (M/Q), different start dates (1946→2009),
22 variants in `WS_TC` alone. A wide table would be a swamp of NULLs and need
`ALTER TABLE` per new series. Tall is the natural SDMX shape and lets us add any series
later by inserting rows. The meta table keeps the opaque BIS unit codes (`770`/`771`/
`628`/`XDC`) self-documenting (unit/frequency are per-series constants — they live in
meta, NOT denormalized onto every observation row).

### Decision 5 — Series scope: **8 core series, monthly/quarterly only (no daily)**
Persist exactly these 8 fully-qualified series (drawn from `scratch/csv_summary.md`):

| series_code | label | unit | freq |
|---|---|---|---|
| `WS_CBPOL.M.IN` | RBI policy rate | % | M |
| `WS_EER.M.R.B.IN` | Real effective exchange rate | index | M |
| `WS_EER.M.N.B.IN` | Nominal effective exchange rate | index | M |
| `WS_XRU.M.IN.INR.A` | USD-INR exchange rate (period avg) | INR/USD | M |
| `WS_TC.Q.IN.P.A.M.770.A` | Credit to private non-fin sector | % GDP | Q |
| `WS_CREDIT_GAP.Q.IN.P.A.C` | Credit-to-GDP gap (Basel) | ppt | Q |
| `WS_DSR.Q.IN.P` | Debt service ratio (private) | % | Q |
| `WS_LONG_CPI.M.IN.771` | CPI inflation (YoY) | % | M |

**Why**: Lijo's pick — lean core covering ~90% of the macro signal. Daily series
(daily `WS_CBPOL`, daily `WS_EER`) are excluded: a daily policy rate is a step function
(~23k repeating rows), and we get market FX elsewhere. The tall table (Decision 4) means
any deferred series (sector-credit splits, property prices, RBI balance sheet, global
liquidity — all confirmed in the recon) can be added later as a one-line registry entry,
**no schema change**.

### Decision 6 — Provenance & revisions: **`source='bis'`, `DO UPDATE` supersede**
`source TEXT NOT NULL = 'bis'`. Upsert `ON CONFLICT (series_code, time_period) DO UPDATE`.

**Why — diverges from V2-018's append-only (intentional)**: Part E confirmed BIS
**revises history retrospectively** — credit, credit-gap, DSR, monthly EER, property
prices, and global liquidity are re-estimated in later releases (only the policy rate and
USD-INR spot are fixed). Append-only (`DO NOTHING`) would freeze stale numbers and rot
the series. So V2-020 follows V2-017's supersede model.

### Decision 7 — Seed window vs. backfill: **rolling ~3-yr seed + one-time full-history backfill**
- **Weekly seed** (`seed-india-macro.mjs`): fetch each series with
  `startPeriod = (currentYear − LOOKBACK_YEARS)` (default **3 years**). Captures new
  observations AND recent revisions while avoiding re-writing decades of unchanged rows
  every week.
- **One-time backfill** (`backfill-india-macro.mjs`): fetch **full history** (omit
  `startPeriod`, or use an inception year) for each series, upsert all. Lijo runs once at
  deploy.

**Why**: BIS returns the whole series in one call (no walked windows, unlike V2-017's
30-day window or V2-019's per-release files), so the backfill is trivial. Revisions
concentrate in recent periods, so a 3-yr rolling seed covers all meaningful revisions;
deep history is seeded once and rarely changes. Mirrors the V2-017 seed+backfill split.

### Decision 8 — Standalone script: **do NOT touch the inherited `seed-bis-data.mjs`**
V2-020 adds a fresh `_bis-india-source.mjs` adapter (re-implementing the ~40-line CSV
parser) + `seed-india-macro.mjs`. It does **not** import from, edit, or extend the
inherited `seed-bis-data.mjs`.

**Why**: Failure isolation — `seed-bis-data.mjs` powers the live WorldMonitor globe; a
shared helper would couple its blast radius to the quant collector. The India series set
diverges (more series, India-only, monthly/quarterly grain), so sharing a fetch function
would mean bending it to two shapes. The ~40-line parser duplication is cheaper than that
coupling. (`seed-bis-data.mjs` is NOT a sacred file, but it's inherited WorldMonitor code
— leave it alone.)

### Decision 9 — Empty/error & prod-execution boundary
- **Per-series resilience**: if one series returns empty/`NaN`-only or a transient
  non-200, log one line and **skip that series, continue**. Only if **all 8** fail does
  the run throw → `runSeed`'s catch → exit 0. Never write a partial/placeholder row.
- **Prod-execution boundary** (per `memory/feedback_v2_prod_execution.md`): this task
  delivers **code + DDL + read-only self-checks only**. **Lijo runs**
  `migrate-india-signals.mjs`, the weekly seed, and the backfill against prod after
  review. The agent does NOT run the scrape or migration against prod, does NOT
  create/run `scratch/` probes (Gemini agent's job). **James implements** from this file.

## Schema (Locked)

Append to the `DDL` template string in `scripts/migrate-india-signals.mjs` (after the
`india_institutional_flows` block) + matching `console.log` lines:

```sql
-- V2-020 BIS India macro rates. Independent non-news macro source: no FK to other
-- tables. TALL design (Decision 4): one row per (series_code, time_period). source +
-- DO UPDATE supersede because BIS revises history (Decision 6). Series identity is the
-- fully-qualified SDMX key; unit/frequency live in india_macro_series_meta, not here.
CREATE TABLE IF NOT EXISTS india_macro_rates (
  series_code  TEXT NOT NULL,            -- '<DATASET>.<full SDMX key>' e.g. 'WS_TC.Q.IN.P.A.M.770.A'
  time_period  TEXT NOT NULL,            -- BIS-native period: 'YYYY-MM' (M) | 'YYYY-Q#' (Q)
  obs_value    DECIMAL(18,6) NOT NULL,   -- the observation (NaN/empty skipped at parse)
  source       TEXT NOT NULL DEFAULT 'bis',
  created_at   TIMESTAMPTZ DEFAULT NOW(),
  updated_at   TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (series_code, time_period)
);
CREATE INDEX IF NOT EXISTS idx_macro_series_period ON india_macro_rates (series_code, time_period DESC);

-- V2-020 series lookup: keeps the opaque BIS codes self-documenting. Refreshed by the
-- seed from the adapter's SERIES registry (the registry is the source of truth).
CREATE TABLE IF NOT EXISTS india_macro_series_meta (
  series_code  TEXT PRIMARY KEY,         -- 'WS_CBPOL.M.IN'
  dataset      TEXT NOT NULL,            -- 'WS_CBPOL'
  sdmx_key     TEXT NOT NULL,            -- 'M.IN' (the key after the dataset)
  label        TEXT NOT NULL,            -- 'RBI policy rate'
  unit         TEXT NOT NULL,            -- '%' | 'index' | 'INR/USD' | '% GDP' | 'ppt'
  frequency    VARCHAR(1) NOT NULL,      -- 'M' | 'Q'
  revises      BOOLEAN NOT NULL,         -- from recon Part E
  updated_at   TIMESTAMPTZ DEFAULT NOW()
);
```

Upsert shapes (seed + backfill):
```sql
-- fact rows
INSERT INTO india_macro_rates (series_code, time_period, obs_value, source)
VALUES ($1, $2, $3, 'bis')
ON CONFLICT (series_code, time_period) DO UPDATE
  SET obs_value = EXCLUDED.obs_value,
      updated_at = NOW();

-- meta rows (refreshed each run from the registry)
INSERT INTO india_macro_series_meta (series_code, dataset, sdmx_key, label, unit, frequency, revises)
VALUES ($1,$2,$3,$4,$5,$6,$7)
ON CONFLICT (series_code) DO UPDATE
  SET label = EXCLUDED.label, unit = EXCLUDED.unit,
      frequency = EXCLUDED.frequency, revises = EXCLUDED.revises,
      updated_at = NOW();
```

```

  1. startPeriod = (currentYear − LOOKBACK_YEARS)         [Decision 7]

       parse CSV → take TIME_PERIOD + OBS_VALUE only       [Decision 2]
       skip NaN/empty obs

  6. return { series, written, latest } → runSeed publishes to status key

backfill-india-macro.mjs (one-time, Lijo runs):
  - same adapter, but full history (no startPeriod / inception year) per series
  - upsert all (DO UPDATE); idempotent, safe to re-run
  - log per-series row counts + final inserted/updated totals
```

## Research Appendix — Source Recon (Gemini agent, 2026-05-21)
*Confirmed from `scratch/csv_summary.md` (11-dataset inventory + Part E) and the 11
`scratch/bis_ws_*.csv` samples. No PENDING cells — recon complete.*

### Source & access (Part E)
- **API host**: `https://stats.bis.org/api/v1/data/<DATASET>/<KEY>?format=csv` — the
  canonical, stable, active SDMX REST host. `data.bis.org` is a human-facing charts
  portal, NOT an API. No auth, no cookies, no warm-up — a standard Chrome `User-Agent`
  is sufficient.
- **Query construction**: SDMX keys are positional and dot-separated. A **fully-qualified**
  key returns exactly one series → parser reads only `TIME_PERIOD` + `OBS_VALUE`.
  ⚠️ **`csv_summary.md` shows 2 series for `WS_CBPOL` (`M.IN` + `D.IN`) and 3 for `WS_EER`**
  because the recon probe left the frequency dimension UNPINNED to inventory everything.
  Our 8 chosen keys are all **fully pinned** (`M.IN`, `M.R.B.IN`, …) → one series each.
  Don't be misled by the multi-series rows in the summary.
- **CSV shape**: BIS returns ALL dimension columns even for a single-series query, so the
  rows carry constant `FREQ`/`REF_AREA`/etc. The parser must read `TIME_PERIOD` +
  `OBS_VALUE` **by header name** and ignore the dimension columns (they're already encoded
  in our `series_code`).
- **History**: `startPeriod=YYYY` (omit `endPeriod`) returns full history to inception in
  one call; no pagination/row cap observed.
- **Licensing / lineage** (Part E): attribute to **"Source: Bank for International
  Settlements (BIS); www.bis.org."** Non-commercial limited extracts allowed free;
  larger/commercial redistribution needs written BIS permission. Store the attribution
  string alongside the data (here: `source='bis'` + this note in the task/lineage record).
  ⚠️ **Product C flag**: BIS terms are stricter than RBI/NSE — any commercial dataset
  resale including BIS series needs BIS permission. Tracked, not a V2-020 blocker.

## Pattern To Follow

`runSeed()` call shape (from `scripts/seed-india-flows.mjs`):

```javascript
#!/usr/bin/env node
import { loadEnvFile, runSeed } from './_seed-utils.mjs';
loadEnvFile(import.meta.url);   // MUST be first

const CANONICAL_KEY = 'news:macro:v1:india';   // STATUS key only — data → PostgreSQL
const CACHE_TTL = 604800;                        // weekly

async function fetchMacro() { /* pipeline flow above; returns summary object */ }
function validate(d) { return typeof d === 'object' && d !== null; }

runSeed('india', 'macro', CANONICAL_KEY, fetchMacro, {
  validateFn: validate, ttlSeconds: CACHE_TTL, sourceVersion: 'macro-bis-sdmx-v1',
```

## Error Scenarios

| Symptom | Likely cause | Fix |
|---|---|---|
| Empty CSV / no rows for a series | Wrong/renamed dataset code or key | Confirm key against `scratch/csv_summary.md`; SDMX keys are positional — match exactly |
| `OBS_VALUE` column missing | Read by wrong header name | Parse by header lookup (`TIME_PERIOD`/`OBS_VALUE`), not by position |
| All series empty / non-200 | Host changed / network | `stats.bis.org/api/v1` is canonical (Part E); if persistently down, all-fail → exit 0 (Decision 9), retry next week |
| Stale old values never update | Used `DO NOTHING` | Use `DO UPDATE` — BIS revises history (Decision 6) |
| Duplicate-key error | Wrong PK / upsert | `ON CONFLICT (series_code, time_period) DO UPDATE` (Decision 4/6) |
| Daily rows appear | Queried a `D.` key | Only the 8 M/Q keys in Decision 5 — no daily |
| Cron overlaps | Missing lock | `runSeed()` provides `seed-lock:india:macro` — keep the wrapper |
